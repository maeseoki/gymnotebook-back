# Auth And Users Compatibility

| Area | Legacy behavior | New behavior | Preserved or changed | Reason | Frontend impact |
| --- | --- | --- | --- | --- | --- |
| Password hashing | Spring Security `BCryptPasswordEncoder`. | New signups use Argon2id. BCrypt hashes are detected by `$2a$`, `$2b$`, `$2y$`, verified, then migrated to Argon2id after successful signin. | Preserved with migration. | Existing users can still sign in while new hashes use stronger parameters. | No client change. |
| Signup role input | Request could contain `role`, but admin/mod cases were commented out and defaulted to `ROLE_USER`. | Signup request is strict and rejects unexpected `role`. User always receives `ROLE_USER`. | Intentionally changed. | Prevents privilege hints and silently ignored security-sensitive fields. | Clients sending `role` during signup get `400 validation_failed`. |
| Signup atomicity | User and roles were saved through JPA in one save operation. | Availability checks, default role load, hashing, user insert, and role insert are inside one Drizzle transaction; unique constraints remain race protection. | Preserved. | Prevents users without default role. | No client change. |
| Signin failure | Spring Security returned one authentication failure. | Unknown username and invalid password both return `401 invalid_credentials`. | Preserved. | Avoids username enumeration. | Clients should use `code`, not message text. |
| JWT payload | Included username-derived subject and roles. | `{ sub, userId, roles }`, where `userId` is immutable numeric id and roles are `ERole[]`. | Extended. | Later use cases avoid username lookups. | Token consumers can continue using `sub`; `userId` is now authoritative. |
| Logout | Cleared Spring Security context in a stateless JWT app. | Kept as stateless compatibility endpoint returning the legacy message. | Preserved as no-op. | JWT revocation is not implemented in this phase. | Clients should discard the token locally. |
| Verify user | `GET /api/user/verifyuser/{username}/{email}` returned legacy Spanish messages. | Legacy path remains. Added `GET /api/user/verifyuser?username=&email=` returning booleans. | Preserved and extended. | Email in path params is awkward; query form is the future-facing shape. | Existing clients keep working; new clients should use query endpoint. |
| User list/current user | Responses contained public user fields and roles. | Responses exclude password hashes; `/me` resolves by JWT `userId`, not username. | Preserved with safer lookup. | Deleted/renamed users are handled by immutable id. | No expected change except stricter missing-user `404 user_not_found`. |
| Role assignment | Admin-only; could assign admin/mod; route code replaced roles by full set. | Admin-only; assigns admin/mod atomically and preserves existing roles; duplicate assignment returns `409 role_already_assigned`. | Preserved with stronger errors. | Business rules moved to use case and stable error codes. | Error status/code may differ from legacy `400` message responses. |
| Role removal | Admin-only; could remove admin/mod. | Admin-only; removes admin/mod atomically; cannot remove `ROLE_USER`; cannot remove final admin. | Intentionally changed. | Avoid accidental lockout and unsafe role changes. | Some legacy removals now return `409 cannot_delete_last_admin` or `400 invalid_role_change`. |
| Delete user | Admin-only; returned `200` message. | Admin-only; returns `204`; prevents self-delete and final-admin delete. | Intentionally changed. | Safer destructive operation and standard REST no-content response. | Clients expecting a message body must tolerate `204`. |
| Error contract | Mixed message-only responses. | Domain errors use stable codes: `invalid_credentials`, `username_already_exists`, `email_already_exists`, `user_not_found`, `role_not_found`, `role_already_assigned`, `role_not_assigned`, `invalid_role_change`, `cannot_delete_self`, `cannot_delete_last_admin`. | Changed. | Clients can branch on `code` consistently. | Frontend should prefer `code` over human message text. |
